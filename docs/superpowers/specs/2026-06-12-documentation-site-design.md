# Documentation site — design

*Approved by Liz 2026-06-12. A standalone documentation SITE (not a dashboard tab),
linked from the top bar like most products. Framework: Astro Starlight (same as the
Knowledge Observatory docs). Lives at `apps/docs/` (docs-as-code). Unfinished sections
are VISIBLE in nav with an explicit "Not yet documented" status — never hallucinated,
never silently blank.*

## Purpose + audience

This documentation is for a standard-setting biological-modeling capability: it is what
regulators, standards bodies, and scientific reviewers will read to understand, trust, and
build regulations around the SDE extraction method. Therefore the bar is **rigor**:

- **Nothing hallucinated.** Every claim traces to something real in the repo (cite it).
- **Blank-until-worthy.** A section that isn't substantiated yet says so explicitly
  ("Not yet documented — filled when the underlying capability is verified"), the same
  discipline as the `Agent Drafts/` provenance rule and the canon's "absence holds."
- **Precise + stepwise + diagrammed.** Match the tone/rigor of Liz's UX_UI_Final report:
  academic, no marketing, requirement/decision tables, labeled diagrams (Mermaid) that
  prove points — not the report's exact walkthrough, just its level of detail.

## Three inputs, three roles (do not mix them)

| Input | Governs |
|---|---|

The categories come from Diátaxis + the project's real domain, NOT from the UX report.

## Diátaxis structure (the nav)

Four top-level categories + decisions, each page exactly one type:

- **Explanation** (understand) — the *why*. The document-architecture canon; present/absent;
  "the document presses, absence holds"; the observability spine; provenance/lineage model;
  the three-pillar system (canon · HITL · confidence).
- **Reference** (look up) — the *what*, exhaustively. The Pydantic extraction schema
  (`FigureExtraction`, `Slot`, present/absent); the database schema; the worker/pipeline
  stages; the lineage/hash model; the targeting modes.
- **How-to** (a goal) — task recipes. Run the worker; apply migrations; configure env;
  add a paper + verify it; deploy.
- **Tutorials** (learn by doing) — guided first walk-throughs (e.g. "extract your first
  paper end-to-end"). Most will be "Not yet documented" until the live pipeline is proven.
- **Decisions (ADRs)** — the load-bearing choices, captured at decision time: OpenAI+Pydantic
  over LangGraph; the canon; single-team RLS; public papers bucket; Astro Starlight; etc.

At least: the 7-transfer observability flow; the pipeline state machine

## Tech

- `apps/docs/` — Astro + `@astrojs/starlight`. Content in `src/content/docs/<category>/`.
- Top bar: title, search (built in), GitHub social link, theme toggle (Starlight defaults).
- Sidebar: the five categories above, autogenerated or explicit order.
- A reusable "Not yet documented" callout (Starlight `<Aside type="caution">`) for status.
- Deploy as a separate site (Vercel), linked from the dashboard's top bar.

## Honest seams

- This is a SKELETON: real pages where substantiated, status banners elsewhere. It will
  fill in slowly as capabilities are verified — that's intended, not a gap.
- Cross-link to the live specs/loom where a page summarizes something fuller.
- ADRs are written at decision time; back-filling the already-made ones is explicitly
  noted as a one-time catch-up, dated.

## Out of scope (now)

- Filling the NOT-YET pages (by design — they wait for the capability).
- Pixel-cloning the Observatory theme (same framework + shape is the goal, not identical CSS).
